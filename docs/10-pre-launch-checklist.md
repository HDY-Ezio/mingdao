## 安全检查清单

### 2.1 认证与授权

- [ ] 所有受保护路由需要认证
- [ ] JWT Token 安全配置
  - [ ] 使用 HttpOnly Cookie（推荐）
  - [ ] 设置合理的过期时间
  - [ ] Secure 标志（HTTPS 环境）
  - [ ] SameSite 配置
- [ ] 密码强度要求
- [ ] 登录失败限流（防止暴力破解）
- [ ] 注销后 Token 失效

### 2.2 数据安全

- [ ] RLS（行级安全）策略配置正确
  - [ ] 所有用户数据表启用 RLS
  - [ ] 策略正确限制用户只能访问自己的数据
  - [ ] 公开数据表（products 等）设置正确
- [ ] 敏感数据加密
  - [ ] 密码使用 bcrypt/Argon2 哈希
  - [ ] 不存储明文密码
- [ ] 输入验证
  - [ ] 所有用户输入都经过验证
  - [ ] 防止 SQL 注入（使用参数化查询）
  - [ ] 防止 XSS 攻击（React 默认转义）

### 2.3 API 安全

- [ ] API 路由鉴权
  - [ ] 需要登录的 API 检查用户身份
  - [ ] 用户只能操作自己的数据
- [ ] 速率限制
  - [ ] 登录接口限流
  - [ ] 支付接口防重复提交
  - [ ] AI 对话接口限流
- [ ] Webhook 安全
  - [ ] PayPal Webhook 验证签名
  - [ ] 不依赖前端传来的金额/状态
- [ ] CORS 配置
  - [ ] 生产环境不使用 `*` 通配符
  - [ ] 只允许必要的域名

### 2.4 环境变量与密钥

- [ ] 所有密钥通过环境变量管理
- [ ] `.env.local` 不提交到 Git
- [ ] 生产环境密钥与开发环境分离
- [ ] 密钥强度足够
- [ ] Service Role Key 只在服务端使用
- [ ] 客户端只暴露 `NEXT_PUBLIC_` 前缀的变量

### 2.5 支付安全

- [ ] 支付金额在服务端验证（不信任客户端）
- [ ] 订单幂等性（防止重复支付）
- [ ] Webhook 签名验证
- [ ] 支付状态以服务端/Webhook 为准
- [ ] 不存储信用卡信息（使用 PayPal 托管）

### 2.6 其他安全

- [ ] HTTPS 全站加密
- [ ] 安全 Header
  - [ ] X-Frame-Options
  - [ ] X-Content-Type-Options
  - [ ] Content-Security-Policy（可选）
  - [ ] Referrer-Policy
- [ ] 错误信息不泄露敏感信息
  - [ ] 生产环境不显示详细错误堆栈
  - [ ] 数据库错误不直接返回给用户

## 回滚预案

### 触发回滚的条件

- [ ] 核心功能完全不可用（注册/登录/支付）
- [ ] 数据安全问题（数据泄露、数据丢失）
- [ ] 严重性能问题（页面加载 > 10s）
- [ ] 大规模用户投诉
- [ ] 第三方服务严重故障且无法快速恢复

### 回滚步骤

1. **确认问题**（5 分钟）
   - 确认问题范围和严重程度
   - 判断是否需要回滚

2. **执行回滚**（5 分钟）
   - Vercel: 进入 Deployments → 选择上一个稳定版本 → "Promote to Production"
   - 或在命令行执行 `vercel rollback`

3. **验证回滚**（5 分钟）
   - 确认旧版本正常运行
   - 确认核心功能恢复

4. **问题排查**（同时进行）
   - 在预发布环境复现问题
   - 定位问题原因
   - 修复后重新部署

5. **通知用户**（如影响较大）
   - 发布状态公告
   - 说明预计恢复时间

### 数据库回滚

- **Supabase**: 可使用 Point-in-Time Recovery（Pro 版及以上）
- 或从最近的备份恢复
- **注意**: 数据库回滚会丢失回滚点之后的新数据，谨慎使用
